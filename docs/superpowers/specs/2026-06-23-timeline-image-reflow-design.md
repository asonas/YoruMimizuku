# タイムライン画像レイアウト改善（縦長クロップ＋幅リフロー）設計

- 日付: 2026-06-23
- 対象: macOS アプリ（`apps/macos`）のタイムライン投稿行 `PostRowView`
- 状態: 設計合意済み（実装計画はこれから）

## 背景と問題

タイムラインの単一画像は `PostRowView.swift` の `singleImage` で、実アスペクト比に
従って描画している。アスペクト比 `ratio`（幅/高さ）は `[0.7, 5.0]` にクランプされる
ため、縦長画像でも下限 `0.7` までしか潰されない。結果、表示高さは最大で
`幅 / 0.7 ≈ 1.43 × 幅` まで伸び、compact(320pt) で約 457pt、comfortable(440pt) で
約 628pt に達する。縦長画像つきの投稿だけ縦に大きく伸び、タイムラインが間延びする。

加えて、メインウィンドウは `NavigationSplitView`（サイドバー＋フィード列）で、フィード列
には最大幅の上限がない（`minWidth: 600` のみ）。ウィンドウを横に広げると投稿行は
ウィンドウ幅いっぱいに伸び、本文テキストは幅いっぱいに折り返される一方、画像は
左寄せのまま固定幅で取り残される。広い画面で横幅が活かされていない。

## 目標

1. 縦長画像で投稿の縦が暴れないよう、表示高さに上限を設けてクロップする。
2. ウィンドウが広いとき、本文を左・メディアを右に並べるリフローで横幅を活用し、
   同時に縦長画像が右列に逃げることで投稿全体の縦を抑える。

非目標: サイドバーや会話ビュー、ライトボックスのレイアウト変更。引用投稿カードや
グリッド画像の内部レイアウトの作り直し。密度設定（compact/comfortable）の意味変更。

## 決定事項（合意済み）

### A. 縦長画像のクロップ（狭い時・広い時の両方に適用）

- 単一画像の表示高さの上限を **5:4（高さ ≤ 1.25 × 表示幅）** とする。
  これはアスペクト比に直すと下限 `ratio = 0.8`。
- 横長側の上限は現状維持（パノラマ対策のため `ratio` 上限 `5.0` は据え置き）。
- 自然アスペクト比が上限より縦長（`naturalRatio < 0.8`）のときは、画像を
  **上寄せ**でクロップして 5:4 の箱に収める（`scaledToFill` + 上寄せ）。
- クロップが実際に発生した場合のみ、下端にフェードのグラデーションと
  「タップで全体表示」のヒントを重ねる。クロップしていない画像には出さない。
- フルサイズはライトボックスで常に確認できる（既存の `openLightbox` を流用）。
- 複数画像グリッド（2列・各 140pt 固定・cover クロップ）は現状維持。

### B. 幅リフロー（本文 左 / メディア 右）

- 基準は「コンテンツ幅」= フィード列の実効幅（ウィンドウ幅 − サイドバー − 行の水平余白）。
- **コンテンツ幅 < 680pt（閾値）**: 現状どおり縦積み（本文 → メディア → カード → 引用 → アクションバー）。
- **コンテンツ幅 ≥ 680pt**: 投稿の `content` 領域を 2 列にする。
  - 左 = 本文列: 著者行、本文テキスト、引用投稿カード、アクションバー。
  - 右 = メディア列: 画像（単一/グリッド）、動画ポスター、リンクカード(OGP) を縦積み。
  - 右列に逃がすのは **画像・動画・リンクカードのみ**。**引用投稿は左の本文列に残す**。
- 寸法:
  - 閾値: **680pt**（= 本文最小 360 + 列間 16 + 右列 300, を丸めた値）
  - 右メディア列: **300pt 固定**、行の**右端に固定（右寄せ）**
  - 列間: **16pt**
  - 本文列: **残り幅いっぱいに伸びる（上限なし）**
- 幅ごとの振る舞い:
  - リフロー時は本文列がウィンドウ幅に合わせて伸び、メディア列（300pt 固定）は常に右端に張り付く。
  - これにより、メディア右の余白も本文-メディア間のギャップも生じない。広げるほど本文の行長が伸びるが、
    右端に取り残される余白を避けることを優先する（2026-06-23 の方針更新。当初は本文上限 620pt・左寄せ・
    右に余白だったが、広い窓で右側に余白が伸びるのを嫌って本文フィル＋メディア右端固定に変更）。

## レイアウト構造

現状の行構造:

```
HStack(top) {
  avatarColumn            // アバター + スレッド線
  content (VStack) {      // 著者行 / 本文 / mediaSection / linkCard / quote / actionBar
    ...
  }
  Spacer
}
```

リフロー後、`content` の中身を幅で 2 パターンに分岐する。メディア（画像・動画・
リンクカード）を 1 つの `mediaColumn` ビューに括り出し、配置だけを切り替える。

- 狭い時（< 680）:

```
content(VStack) {
  authorLine
  bodyText
  mediaColumn          // 画像 → 動画 → リンクカード（縦積み）
  quoteCard            // 引用は従来どおり本文の下
  actionBar
}
```

- 広い時（≥ 680）:

```
content(HStack top, spacing: 16, maxWidth: .infinity) {
  VStack(textColumn, maxWidth: .infinity) {   // 残り幅いっぱいに伸びる
    authorLine
    bodyText
    quoteCard          // 引用は左（本文列）に残す
    actionBar
  }
  mediaColumn(width: 300)   // 右端に固定。画像 → 動画 → リンクカード（縦積み）
}
```

`mediaColumn` は配置先に応じて最大幅が変わる（狭い時は `imageMaxWidth`、広い時は
右列幅 300pt）。`singleImage` / `imageGrid` / `VideoPosterView` / `LinkCardView` は
渡された幅を上限として描画する。アクションバーと引用は常に左（本文列）側。

## 幅の測定方針

各行で `GeometryReader` を多数生成すると `List` のスクロール性能に響くため、
**フィード列で 1 回だけ実効幅を測り、Environment 経由で各行に渡す**。

- `FeedView` のコンテンツ領域で `onGeometryChange`（または 1 つの `GeometryReader`）
  により幅を取得し、`EnvironmentValues` の新規キー（例: `timelineContentWidth`）へ注入する。
- `PostRowView` は環境値から自身の `content` 領域の使用可能幅を算出する
  （`コンテンツ幅 − 水平余白(12/16) − アバター列幅 − 列間`）。その値と閾値 680pt を
  比較してレイアウトを分岐する。
- 既存の `imageMaxWidth`（compact 320 / comfortable 440）は狭い時の上限として維持。

## エラー処理・エッジケース

- `aspectRatio` が `nil` の画像: 従来どおり 4:3 を仮定。縦長判定にもこの値を使う。
- 画像の読込中/失敗: 既存の `imagePhaseContent`（プレースホルダ / 失敗アイコン）を流用。
  クロップ箱のサイズは確定済みなのでレイアウトは揺れない。
- リンクカードが右列 300pt で窮屈な場合: `LinkCardView` は横並びサムネを縮める既存の
  挙動に委ねる（最小幅の破綻が出るなら実装時に縦積みフォールバックを検討、ただし本設計の
  既定は既存 `LinkCardView` をそのまま使う）。
- 複数画像グリッド（2列）が右列 300pt に入る場合: 列あたり約 (300 − 5) / 2 ≈ 147pt。
  各タイル高さ 140pt 固定は維持。
- センシティブメディアのぼかし（`mediaWarning`）: `mediaColumn` 全体に対して従来どおり
  適用する（配置が左下でも右でも同じ）。
- 動画と画像が同居する投稿: 右列に画像 → 動画ポスターの順で縦積み（現行の `mediaSection`
  の順序を踏襲）。

## テスト方針

ロジック（純粋関数）に切り出してユニットテスト可能にする。

- レイアウト判定関数: `(コンテンツ幅) -> .vertical | .reflow` が閾値 680 を境に切り替わる
  （679→vertical, 680→reflow）。
- 縦長クロップ計算: `(naturalRatio, 表示幅) -> 表示高さ` が 5:4 で頭打ちになる
  （`ratio=0.45, 幅=300 → 高さ 375`、`ratio=1.0 → クロップなし`、横長は現状維持）。
- クロップ発生フラグ: `naturalRatio < 0.8` のときだけ「全体表示」ヒントを出す。
- （本文列幅の上限は廃止。リフロー時は本文がフィルし、幅計算は不要になった。）

これらは `YoruMimizukuKit` 側の純粋ヘルパー（例: `TimelineLayout`）に置き、
`YoruMimizukuKitTests` でテストする。View 本体（SwiftUI）は薄く保つ。

## 段階的な進め方（TDD 想定）

1. 縦長クロップのロジックとヘルパーを追加（Red→Green）。`singleImage` を 5:4 上限に。
2. クロップ時のフェード＋「タップで全体表示」オーバーレイ。
3. レイアウト判定ヘルパー（閾値 680）と Environment 幅注入。
4. `content` を `mediaColumn` 括り出し＋幅で 2 パターン分岐（本文左 / メディア右）。
5. 寸法（右列 300 / 列間 16）。本文はフィルし、メディア列は右端固定（右に余白を作らない）。

各ステップでコア package のテストを回し、最後にアプリをビルドして実機の見た目を確認する。
