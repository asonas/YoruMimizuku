# フィードのスレッド束ねを自己スレッドのみに限定する設計書

- 日付: 2026-07-03
- ステータス: 設計合意済み（実装計画はこの後 `writing-plans` で作成）
- 作業ブランチ: `feature/feed-self-thread-only`
- 関連: `docs/wiki/behaviors/timeline-streaming.md`（フィードのスレッドグループ化）、`2026-06-08-yorumimizuku-timeline-ux-enhancements-design.md`

## 1. 背景と問題

タイムライン（`FeedView`）は `FeedThreading.arrange` で同一スレッドの投稿を束ねて表示する。この仕組みは Bluesky web を模したもので、本来は「同じ人が自分の投稿に連続でぶら下げた自己スレッド（1/3 … 3/3）」を1ブロックにまとめることを意図している。

しかし現在の `groupKey` は、`replyParent` を**著者を問わず無条件で最上位の祖先まで**辿ってグループキーとする。その結果、同じ root に複数人が返信して枝分かれした会話まで、すべてが1つのグループに束ねられる。グループ内のメンバーは `createdAt` 昇順にソートされ、横インデントなし・縦のコネクタ線のみの1列で描画されるため、複数人・枝分かれの会話が「時系列に並んだ1列のフラットなリスト」に畳まれ、投稿の親子・枝分かれ構造が完全に失われる。

### 実データによる再現（DDJ-FLX4 スレッド）

`app.bsky.feed.getPostThread` が返す実際のツリー構造:

```
root: kdmsnr「おじさんの趣味にしては高いけどDDJ-FLX4買うかー」(01:39)
├─ takkanm「ギターに比べたら圧倒的に安くないです？」(04:57)
│   └─ kdmsnr「ギターやらないし」(05:19)
│       └─ takkanm「弾けないギターを弾くんですよ」(05:20)
├─ kdmsnr「蕎麦を打ち始めるよりよさそう」(01:39)
└─ asonas「使ってないDDJ-FLX4があるんですが…」(04:21)
    └─ kdmsnr「わーありがとうございます！…」(05:18)
```

現状のフィード表示（`FeedThreading.arrange` の出力）は、これらすべてを root 起点の1グループに束ね、createdAt 昇順で:

```
蕎麦 → 使ってない → ギターに比べたら → わーありがとう → ギターやらないし → 弾けない
```

という完全な時系列フラットの1列になる。誰が誰に返信しているのかが読み取れない。

注: コア（`ThreadNode.childTree`）と会話タブ（`ConversationView`）は、この実データを depth 付きの正しいツリーとして構築・表示できることを確認済みである。壊れているのはフィード側のグループ化ロジックのみ。

## 2. ゴール / 非ゴール

### ゴール
- フィードで複数人・枝分かれの会話が1列のフラットなブロックに畳まれる挙動をなくす。
- 束ねる対象を「同一著者の連続する自己スレッドチェーン」のみに限定する。
- 他者への返信・他者からの返信はグループ境界で切り、それぞれ独立した行に戻して既存の「@X への返信」文脈ラベルで親子関係を読めるようにする。

### 非ゴール
- 会話タブ（`ConversationView`）のインデント表現の変更（既にツリー表示できており、本件のフィード問題とは別件）。
- フィード内でのツリーインデント表示（検討したが不採用。理由は §5）。
- 公式アプリのような「返信ごとに直近の親をプレビュー表示するペア方式」への作り替え（検討したが不採用。理由は §5）。
- `PostRowView` / `FeedView` の表示ロジックの変更（本設計では変更不要）。

## 3. 変更方針

`FeedThreading.arrange` の `groupKey(for:)` における `replyParent` の遡上条件を変更する。

- 現状: `current.replyParent?.post.id` が page 上に存在する限り遡上する（著者を問わない）。
- 変更後: `current.replyParent?.post.id` が page 上に存在し、**かつ「親の著者 == current の著者」の間だけ**遡上する。著者が変わる境界で遡上を止め、そこがグループキーになる。

これにより、束ねられるのは「同じ人が自分の投稿にぶら下げ続けた自己スレッド」だけになる。他者が絡んだ時点でグループが切れる。

`groupKey` 以外（メンバーの `createdAt` 昇順ソート、最古参メンバー位置での emit、重複 ID の一度きり emit、`connectsToPrevious/Next` の付与、サイクルガード）は現状のまま維持する。自己スレッドは本質的に線形なので、`createdAt` 昇順ソートで正しく並ぶ。

### 著者一致の判定

`PostDisplay` が公開している著者識別子は `authorHandle` のみ（著者 DID の専用フィールドは持たない）。同一著者判定は `authorHandle` の一致で行う。

判定式（概念）: `current.authorHandle == parentPost.authorHandle` が真の間だけ遡上する。

- 1回のフィード取得（1ページ）の中では、同一人物の投稿は必ず同じハンドルを持ち、別人が同じハンドルを共有することはない。したがってページ内グループ化の文脈では `authorHandle` は「同一著者」の正確な判別子になる。ハンドルがセッション途中で変わるケースはフィードのグループ化には影響しない。
- DID 比較（`ATURI.repo(id)`）も検討したが、(a) `PostDisplay` は DID フィールドを持たず `id`（AT-URI）からの抽出が必要で `BlueskyCore` への依存追加を伴う、(b) 既存テストは投稿 ID に AT-URI ではない単純文字列を使うため `ATURI.repo` が nil になり破綻する、という理由で不採用。`authorHandle` 比較なら `FeedThreading.swift` の import 追加も不要で、既存テストとも整合する。

## 4. 表示の帰結（DDJ-FLX4 スレッド）

変更後、フィードは以下のように表示される。

- `{root, 蕎麦}`（どちらも kdmsnr、root への自己返信）→ 1ブロック（アバター間コネクタ線、ブロック内 divider 省略）
- takkanm「ギターに比べたら」→ 独立行・「@kdmsnr への返信」ラベル
- kdmsnr「ギターやらないし」→ 独立行・「@takkanm への返信」ラベル
- takkanm「弾けない」→ 独立行・「@kdmsnr への返信」ラベル
- asonas「使ってない」→ 独立行・「@kdmsnr への返信」ラベル
- kdmsnr「わーありがとう」→ 独立行・「@asonas への返信」ラベル

各返信に「誰への返信か」が表示され、親子関係が読める。`PostRowView` は既に `showReplyMarker && !connectsToPrevious && post.replyParent != nil` のときこのラベルを描画するため、独立行に戻った返信は自動的に文脈ラベルを得る。`FeedView` / `PostRowView` 側の変更は不要。

### エッジケース

- **同一著者が同じ親に複数の返信をぶら下げた場合**（同一著者の兄弟返信）: どちらも著者一致で同じ祖先まで遡上するため、同一グループに入り createdAt 昇順で並ぶ。これは自己スレッド的な扱いとして許容する（線形チェーンではないが同一著者であり、実害は小さい。今回の対象スレッドでは発生しない）。
- **親が page 上に存在しない返信**: 現状どおり、その投稿はグループキー＝自分自身となり独立行のまま。文脈ラベルは `replyParent` が hydrate されていれば表示される。
- **`authorHandle` が空文字などの想定外**: 親と厳密一致しなければ遡上を止める（グループが切れる方向＝安全側）。空ハンドル同士がたまたま一致するケースは実運用では発生しないが、発生してもフラット畳み込みは起きず独立行寄りになるだけで実害はない。

## 5. 検討した代替案と不採用理由

- **フィード内でツリーをインデント表示（Option B）**: 現状のグループ化を維持しつつ depth でインデントし枝分かれを見せる案。会話タブに近い見た目になるが、フィードがツリー化して横に広がりやすく、既存のフィード設計思想（1列＋コネクタ線）から外れる。不採用。
- **親post＋返信のペア方式（Option C）**: 各返信の直上に直接の親だけを文脈プレビューとして出し、兄弟返信は別々の行に。公式アプリのフィードに最も忠実だが、`FeedThreading` の出力モデル（グループ単位）をペア＋親プレビュー単位に作り替える必要があり重い。今回のスコープでは過剰。不採用。

## 6. テスト（TDD）

`core/Tests/YoruMimizukuKitTests/FeedThreadingTests.swift` にケースを追加・調整する。Red → Green → Refactor を1ステップずつ進める。

1. **複数人の枝分かれ返信は束ねられない**: root（著者A）＋ 著者B の返信 ＋ 著者C の返信、のような入力で、B・C の返信がそれぞれ独立した `ThreadedFeedItem`（`connectsToPrevious == false`）になることを検証。今回の DDJ スレッドの縮図。
2. **同一著者の自己スレッドは従来どおり束ねられる**: 著者A の root ＋ A の自己返信 ＋ さらに A の自己返信、が1ブロック（`connectsToPrevious/Next` が連結）にまとまり、createdAt 昇順に並ぶことを検証。
3. **混在**: 著者A の root＋A の自己返信は束ね、そこにぶら下がる著者B の返信は独立行に切れることを検証（境界で著者が変わるとグループが切れる）。
4. **既存テストの回帰**: 現行の `FeedThreadingTests` は全ケースが単一著者（`a.example`）を用いており、多著者チェーンを1グループとして期待するテストは存在しない。したがって本変更後も既存の自己スレッド系テストはそのまま green で通るべきで、期待値の更新は不要（もし想定外に落ちるケースがあれば、それは本設計の意図どおりの挙動変更なので個別に判断する）。新規の多著者テストでは、`post` ヘルパーに著者（ハンドル）を指定できる引数を追加して別著者を表現する。

サイクルガード・重複 ID・親不在の既存挙動が壊れていないことも既存テストで担保する。

## 7. 影響範囲

- 変更: `core/Sources/YoruMimizukuKit/FeedThreading.swift`（`groupKey` の遡上条件のみ。import 追加は不要）。
- 変更: `core/Tests/YoruMimizukuKitTests/FeedThreadingTests.swift`（テスト追加・期待値更新）。
- 変更なし: `apps/macos/Views/FeedView.swift`、`apps/macos/Views/PostRowView.swift`（既存の `connectsToPrevious/Next` とラベル機構をそのまま利用）。
- iPadOS / Windows: どちらも同じ `FeedThreading.arrange` を共有（Windows は `yoru_feed_arrange` ブリッジ経由）しているため、本修正は自動的に全プラットフォームへ波及する。追加のプラットフォーム別コード変更は不要。
- ドキュメント: `docs/wiki/behaviors/timeline-streaming.md` と `docs/wiki/support-matrix.md` のフィードスレッドグループ化の記述を、「最上位祖先で束ねる」から「同一著者の自己スレッドのみ束ねる」に更新する（`wiki-update` スキルで実施）。
