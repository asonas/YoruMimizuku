# YoruMimizuku v1.0.0 ロードマップ（ギャップ分析と優先順位）

- 日付: 2026-06-11
- ステータス: スコープ合意済み（2026-06-11 ユーザー決定を反映）
- 現在のバージョン: v0.8.0（build 11、安定版リリース済み）
- 目的: 設計書 `2026-06-04-yorumimizuku-design.md` が定義する v1 スコープと現実装の差分を列挙し、v1.0.0「安定版としての完成」までの作業を優先順位付きで計画する

## スコープ決定（2026-06-11）

ギャップ分析に基づき、以下のとおり合意した。

- **v1.0.0 に含める**: A-5（アカウントセレクタ/ログアウト）、B-1（引用ポスト表示）、B-2（動画ポスター）、B-3（削除 UI）、B-4（通知設定）、B-5（エラー UX）。中間リリース 0.9.0 は設けず、これらをまとめて v1.0.0 とする（途中経過は従来どおり dev チャンネルで配布してよい）
- **v1.x 系に先送り**: A-4（OS 通知バナー + Dock バッジ）、A-1（カスタムフィードタブ）、A-2（リストタブ）
- **Jetstream（A-3）は見送り**: スケジュールに載せない。設計書 §6.2 のフォールバック規定（ポーリング）を恒常の動作モードとして v1 を成立させる。設計書にこの決定を追記し、wiki にも記載する

## 調査方法

設計書 §2（v1 のゴール）の各項目を、`core/Sources/` と `apps/macos/` の実コードに対して突き合わせた。wiki（`docs/wiki/`）の記載は一部実態より先行していたため、コード grep で裏取りした（後述 C 節）。macOS が一次ターゲットであるため、本ロードマップの完成条件は macOS 基準とする。Windows / iPadOS のパリティは v1.0.0 のブロッカーにしない（D 節）。

## A. 設計書 v1 スコープのうち未実装（macOS）

### A-1. カスタムフィードタブ（`app.bsky.feed.getFeed`）

設計書 §2 / §7.3 は 7 種のタブソース（home / notifications / custom feed / list / author / search / thread）を v1 スコープと定める。現状、custom feed は XRPC サービス（`core/Sources/BlueskyCore/XRPC/` に FeedService が無い）も、タブ種別（`WorkspaceTab` enum は home / notifications / filter / conversation / author のみ）も、タブ追加 UI も存在しない。

- 作業: `FeedService`（getFeed + ピン留めフィード一覧 `getPreferences` or `app.bsky.feed.getActorFeeds` / `getSuggestedFeeds` の最小形）、`WorkspaceTab.feed(UUID)`、サイドバーのタブ追加 UI、`TimelineLoading` 準拠のローダ
- 規模: 中（既存の author タブ追加と同型のパターンが使える）

### A-2. リストタブ（`app.bsky.graph.getListFeed`）

A-1 と同様にサービス・タブ種別・UI のすべてが無い。

- 作業: `ListService`（getListFeed + 自分のリスト列挙 `app.bsky.graph.getLists`）、`WorkspaceTab.list(UUID)`、タブ追加 UI、ローダ
- 規模: 中（A-1 と同時にやると追加 UI を共通化できる）

### A-3. Jetstream ライブ更新（home / list）

設計書 §2.3 / §6 の中核項目だが完全に未実装。WebSocket ポートも Jetstream デコーダも watchdog も存在せず、全ソースがポーリングで動いている（コード内の "Jetstream" は preview 用フィクスチャ文字列 1 箇所のみ）。なお現状のポーリングは実用上は機能しており、§6.2 が「上限超過時はポーリングにフォールバック」を認めている。

- **決定（2026-06-11）: 見送り。** ポーリングを恒常の動作モードとして v1 を成立させる。将来再検討する場合は独立した設計スペック + 実装プランを書く（規模: 大）

### A-4. OS 通知バナー + Dock バッジ

設計書 §2.5 / §9.2 のスコープ。アプリ内の未読バッジ（サイドバー）は実装済みだが、`UNUserNotificationCenter` バナー・Dock バッジ・通知権限リクエストは macOS アプリに一切存在しない（grep で UNUserNotification / dockTile ともゼロ件）。設計書が 6 つの OS ポートの 1 つに挙げる「OS 通知」ポート自体が未定義。

- 作業: OS 通知ポート（protocol）+ Apple アダプタ、バックグラウンドポーリングからの新着検出（既存 `UnreadCounter` 基盤を流用）、バナー発火、Dock バッジ更新、初回権限リクエスト、設定画面に通知タブ（有効/無効、ポーリング間隔）
- 規模: 中

### A-5. アカウントセレクタ UI / ログアウト

設計書 §7.1 / §8 は「ウィンドウ右上のアカウントセレクタで per-window にアカウントを切替」と定める。コア（`AccountManager`、per-DID セッション、`remove()`）は実装済みだが、UI が無い。現状アカウントが切り替わるのはセッション失効時のフォールバックだけで、ユーザーが能動的に切替・ログアウト・アカウント追加（2 人目以降）する手段がない。

- 作業: サイドバーのアカウントフッター（既存）をメニュー化: アカウント一覧 + 切替、アカウント追加（ログインフロー起動）、ログアウト（remove + Keychain 掃除）
- 規模: 小〜中

## B. 実装済みだが不完全（安定版として品質不足）

### B-1. 引用ポスト（record embed）がタイムラインで丸ごと欠落

`PostEmbed` は images / external 以外（record / recordWithMedia / video）を「空」にデコードして捨てる（`core/Sources/BlueskyCore/Models/Timeline.swift`）。つまり他人の引用ポストを見ると、引用コメントだけが表示され引用先が無言で消える。日常利用で最も目立つ不完全さで、ユーザーは「壊れている」と感じる。

- 作業: `app.bsky.embed.record#view`（+ recordWithMedia）の tolerant デコード（notFound / blocked / detached バリアントは捨てる）、`PostDisplay.quotedPost` へのマッピング、PostRowView 内の引用カード（既存 LinkCard と同系のボーダー枠、author 行 + 本文 + サムネ、クリックで会話タブ）
- 規模: 中

### B-2. 動画 embed（`app.bsky.embed.video#view`）が無表示

B-1 と同じ箇所で video も捨てられる。動画付きポストは本文だけが出る。インライン再生は v1.0.0 に必須ではないが、最低限「サムネイル（playlist の poster）+ 再生アイコン + クリックでブラウザ/bsky.app を開く」が無いと欠落に見える。

- 作業: video view のデコード（thumbnail / playlist / aspectRatio）、ポスター画像 + 再生バッジ表示、クリックでポストの permalink を開く
- 規模: 小〜中（インライン再生は post-1.0）

### B-3. 自分のポストの削除 UI が無い

`PostService.deleteRecord()` はコアに実装済みだが UI が無い。誤投稿を消せないクライアントは安定版として成立しない。

- 作業: 自分のポストの行（authorDID == currentDID）にコンテキストメニュー/アクション「削除」、確認ダイアログ、削除後のリストからの除去
- 規模: 小

### B-4. 通知関連の設定が無い

設定画面は外観 / フォント / 表示 / アップデートの 4 タブのみ。設計書 §9.2 の「ポーリング間隔設定」を含め、通知の挙動を制御する UI が無い。v1.0.0 ではアプリ内通知（ポーリング間隔・バッジ表示の有無）の設定を実装する。OS バナーの有効/無効トグルは A-4（1.x）と一緒に追加する。

### B-5. エラー UX（オフライン / レート制限）

トークン失効は処理済みだが、オフラインやレート制限は汎用エラー表示に落ちる。最低限「再試行」導線と、429 のバックオフがあれば v1.0.0 として十分。

- 規模: 小（ポーリングループのバックオフは既存。表示の整備が主）

## C. wiki の過大記載の修正（即時、コード変更なし）

wiki の behavior ページ 2 箇所が、設計書の記述を実装済みであるかのように `macos: full` と記載している。実態に合わせて直し、`mise run wiki:matrix` で support-matrix を再生成する。

1. `docs/wiki/behaviors/timeline-streaming.md`: 「Jetstream live updates (home / list)」 macos: full → **none**（全ソースがポーリング。本文の「Home / List (Jetstream live)」節も「設計上の予定」である旨に書き直す）
2. `docs/wiki/behaviors/notifications.md`: 「OS banner + unread badge」 macos: full → **limited**（アプリ内サイドバーバッジのみ。UNUserNotificationCenter / Dock バッジ未実装である旨を本文に明記）

A-4 が実装されたら（1.x）full に戻す。A-3 は見送りのため、wiki には「設計のみ・見送り」の状態を記録として残す。

## D. v1.0.0 のスコープ外（明示的に先送り）

- **Windows / iPadOS のパリティ**: support-matrix の × 項目（リンクカード、スレッドグルーピング、会話ツリー等）。v1.0.0 は macOS 基準とし、他プラットフォームは追従リリースで埋める
- **モデレーション**: 設計書 §13 が「ラベル詳細設定」を明示的に延期。ミュート/ブロックの操作 UI も post-1.0。ただし**コンテンツラベルによる NSFW 画像のぼかし**は「安定版を人に勧められるか」に直結するため、v1.0.0 の仕上げ項目として残す（やらないと判断したらリリースノートに既知の制限として書く）
- **DM、動画アップロード、granular OAuth scopes**: 設計書 §13 どおり対象外
- **インライン動画再生**: B-2 のサムネ対応まで。再生は post-1.0

## マイルストーン（2026-06-11 合意版）

中間の 0.9.0 安定版は設けない。実装中の途中経過は従来どおり dev チャンネル（1.0.0-dev.N）で配布し、下記 v1.0.0 の項目がすべて揃った時点で v1.0.0 を切る。各機能は着手時に superpowers の設計 + 実装プラン（既存の phase-a〜d と同じ粒度）を別途作成する。

### v1.0.0 — 表示と操作の完全性 + 品質仕上げ

実装順は体感インパクトと依存の少なさから次を推奨する。

1. B-1 引用ポスト表示（record / recordWithMedia）
2. B-2 動画 embed のポスター表示
3. B-3 自分のポスト削除 UI
4. A-5 アカウントセレクタ / ログアウト UI
5. B-4 設定画面の通知タブ（アプリ内ポーリング間隔・バッジ）
6. B-5 エラー UX の整備（オフライン/レート制限の表示と再試行導線）
7. 仕上げ: NSFW ラベルぼかしの実施判断（やらない場合は既知の制限として文書化）、全 behavior wiki の実態再点検（lint + matrix）、回帰テストの通し確認、リリースノートに v1 スコープの達成状況を記載

### v1.x — 先送りした機能

1. A-4 OS 通知バナー + Dock バッジ + 権限リクエスト（+ 設定の OS バナートグル）
2. A-1 カスタムフィードタブ / A-2 リストタブ（タブ追加 UI をフィード/リスト/ユーザー/検索で統合）

### 見送り（スケジュールに載せない）

- A-3 Jetstream ライブ更新一式。ポーリングが恒常の動作モード。再検討時は専用の設計スペックから始める

## 順序の根拠

B-1〜B-3 を先頭に置くのは、引用ポスト欠落と削除不能が「毎日触って一番痛い」既存機能の穴であり、いずれも規模が小〜中で確実に回収できるため。A-5 と B-4 は UI 追加が中心で互いに独立。B-5 と仕上げ項目は他の実装が出揃ってからまとめて検証するのが効率的なので最後に置く。OS 通知とタブ 2 種は設計書の v1 スコープからは外れるが、ユーザー決定（2026-06-11）により 1.x 系へ送る。設計書 §2 との差分（Jetstream 見送り、OS 通知・カスタムフィード・リストの 1.x 送り）は設計書末尾の追記と wiki に記録する。
